CLI First

CLI Architecture

目标不是做一个“能聊天的终端”，而是做一个接近 Codex / Claude Code 的产品级 CLI。 真正决定上限的是输入系统、PTY、事件循环和状态架构，不是模型接入本身。

推荐结论： 直接走 Rust core + crossterm + ratatui + ratatui-textarea + tokio + portable-pty。

目标

  • 做出接近 Codex / Claude Code 体验的 CLI。
  • 重点解决输入体验、流式交互、工具执行、终端稳定性。
  • 为后续扩展桌面端保留空间，但不让桌面端反过来绑架 CLI 架构。

这里不讨论模型供应商，也不讨论具体视觉风格。核心问题只有一个： 如果要把 CLI 做到产品级，技术架构应该怎么选。

这套选择的理由很直接：CLI 的上限主要由输入系统、终端控制、流式渲染和 PTY 能力决定， 而这些点上，Rust 方案最适合长期打磨成稳定产品。

为什么不要先从前端式 UI 框架出发

如果目标只是快速做一个能聊天的终端界面，很多栈都够用。但如果目标是接近 Codex / Claude Code，输入框不能只是“接收字符串”， 必须被当成一个真正的编辑器子系统；终端也不能只是“渲染文本”，必须被当成一个长期运行的交互工作台。

  1. 输入编辑器
  2. PTY
  3. 事件循环与屏幕渲染
  4. 会话状态与工具运行时
  5. 模型接入

很多项目会反着做，先把模型接口接上，再补终端体验。最后几乎都会卡死在输入层。

推荐分层

1. Core

负责会话状态、消息模型、工具运行状态、配置解析、权限决策、取消和 transcript 持久化。

要求是：不依赖具体 TUI 框架，不直接操作终端，可以被 CLI、桌面端甚至未来其他壳层复用。

2. Terminal Shell

负责 raw mode、键盘事件、焦点切换、屏幕 diff 渲染、布局、滚动和状态栏。

这一层不要承载业务逻辑，只消费 Core 状态并渲染。

3. Editor

这是最关键的一层，负责多行输入、光标移动、选择、删除、撤销重做、history、粘贴和快捷键绑定。

不要把输入框写成几个 string.slice()。应把它视为独立组件，维护自己的 buffer、cursor、selection 和 viewport。

4. Runtime

负责流式模型输出、后台任务、取消、并发工具执行、队列调度和事件总线。建议统一基于 tokio。

5. PTY / Tool Runner

负责 shell 命令执行、交互式子进程、实时 stdout / stderr、ANSI 保留、terminal size 同步和 signal 转发。

这里不要用“child process + pipe”冒充 PTY。

6. Storage

负责用户配置、agent / model 配置、session transcript、command logs、checkpoints 和本地缓存。

建议组合使用 settings.json 与 SQLite。

输入系统设计建议

如果目标是顶级 CLI 体验，输入系统至少要支持这些能力：

  • Enter 发送
  • Shift+Enter 或可配置换行策略
  • Ctrl+J / Ctrl+M 等兼容映射
  • 左右上下移动、按词移动、行首行尾
  • 删除前一个字符、后一个字符、前一个词
  • history
  • 多行粘贴
  • 长文本输入时的 viewport 滚动
关键现实： Shift+Enter 在不同终端里的行为并不完全统一，所以不能把它设计成唯一换行手段。

消息区与 PTY 设计建议

消息区

  • streaming 中的增量渲染
  • 自动滚动与用户手动滚动并存
  • new messages below 提示
  • tool call 折叠
  • stdout / stderr 分流展示
  • metadata 展示
  • error block
  • 可复制 code block

PTY

  • resize 同步
  • 保留 ANSI
  • 交互式进程
  • 中断和退出码管理
  • 超时和 kill policy
  • 输出截断与全文日志分离

不要把 tool 执行结果只当纯字符串，更合理的是给 UI 结构化摘要，把原始日志存进 storage。

配置策略

用户配置

  • provider
  • model
  • api key 引用方式
  • prompt defaults
  • key bindings
  • UI 偏好

系统配置

  • 日志级别
  • 本地目录结构
  • checkpoint 策略
  • transcript 保留策略
  • feature flags

用户配置要稳定、可读、可迁移；系统配置要内部托管，避免把运行噪音暴露给用户。

技术选型对比

首选：Rust

适合长期产品化、强 PTY、强终端控制、可复用 core、可打包分发。

备选：Python + prompt_toolkit

适合最快验证强输入体验原型，但后期工程化分发不如 Rust 干净。

备选：Go + Bubble Tea

适合开发效率与终端产品感之间取平衡，但输入系统上限略弱。

不优先：TypeScript + Ink

做 demo 没问题，但如果目标是顶级 CLI，核心输入层不应继续压在 Ink 上。

分阶段落地

Phase 1

  • 单会话
  • 流式回复
  • 稳定多行输入
  • PTY 执行命令
  • transcript 持久化

Phase 2

  • tool call
  • approvals
  • session history
  • checkpoints
  • background tasks

Phase 3

  • 输入键位配置
  • 更强滚动和搜索
  • message fold / expand
  • rich logs
  • theme
  • 可恢复工作区状态

最终建议

如果目标是“做一个非常好用、长期可维护、未来还能扩桌面端的 CLI”，直接走： Rust core + crossterm + ratatui + ratatui-textarea + tokio + portable-pty。

如果目标是“先最快做出最强输入体验原型”，走： Python + prompt_toolkit。

如果只是因为当前项目是 TypeScript，就继续强绑在 Ink 路线上，这个决定本身会限制产品上限。
